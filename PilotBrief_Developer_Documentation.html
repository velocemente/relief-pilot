PilotBrief — Developer Documentation

Version: v1.9.7  |  Date: April 2026  |  File: index.html

PilotBrief is a single-file HTML/JS progressive web app (PWA) for Atlas Air and Polar Air Cargo relief and augmented crew. It parses Operational Flight Plan (OFP) PDFs and generates structured three-phase preflight briefing notes. No server, build step, or login is required. All data stays on the device.

⚠ Certification Status: PilotBrief is NOT certified aviation software. It is an informational tool only. All parsed data must be independently verified against the official OFP and dispatch release.

2. State Model (S object)

All parsed and user-entered data lives in the global S object. Serialized to localStorage via saveFlight(). PDF blobs stored separately in IndexedDB (SW never touches either store).

4. OFP Parser — Regex Reference

parseOFP(rawText) runs against the full extracted OFP text. All timing data is extracted from self-contained single-line headers. The Atlas/Polar format places ICAO, coordinates, and time on one line — no multi-line look-ahead required.

Core Helper: parseAptLine(keyword)

// Format: ORIG VHHH/HKG HONG KONG INTL Q N22185E113549 0510Z
function parseAptLine(keyword) {
  const re = new RegExp('^' + keyword + '\\s+([A-Z]{4})\\/\\S+[^\\n]*?(\\d{4})Z\\s*$', 'm');
  const m = t.match(re);
  return m ? { icao: m[1], hhmm: m[2] } : null;
}

5. Fuel & Weight Logic

Atlas/Polar OFPs express fuel in raw integer format (lbs×10 or kgs×10). The weight unit is parsed from the OFP narrative header — S.weightUnit is either 'KGS' or 'LBS'.

TOGW Margin Color Coding

margin = togwLimit - togwActual
if (margin > 1000)    → GREEN   (comfortable)
if (margin 500–1000)  → YELLOW  (advisory)
if (margin < 500)     → RED     (near structural limit)

Ballast Fuel

Parsed from OFP ballast section. Displayed as a yellow-highlighted caution row below FRES when present. Does not affect TOGW calculation — crew reference only.

6. ETA Recalculation & OFF Derivation

recalcAllETAs() — OUT-based (planned)

Called when crew edits outZ. Uses the OFP-derived offset (which includes taxi and startup time, unlike the TRIP field).

function recalcAllETAs() {
  const baseMin = hhmm2min(S.outZ);
  if (S.destEtaOffsetMins !== null)
    S.etaDest = addMin(baseMin, S.destEtaOffsetMins);  // OUT + OFP block-time offset
  S.altData.forEach(a => {
    a.currentEta = addMin(baseMin, a.ofpOffsetMins);
  });
}
// Why not TRIP? TRIP = 13h17m (flight time only); OFP offset = 14h17m (incl. taxi)
// Using TRIP gives ETA 18h34Z not the planned 19h34Z.

recalcAllETAsFromOff() — OFF-based (actual)

Called when crew enters actual wheels-off time. Uses tripMinutes for DEST (pure flight time from wheels-up). Subtracts derived taxi time from alternate offsets.

function recalcAllETAsFromOff(offZval) {
  const baseMin = hhmm2min(offZval);
  if (S.tripMinutes !== null)
    S.etaDest = addMin(baseMin, S.tripMinutes);
  const taxiMins = S.destEtaOffsetMins - S.tripMinutes;  // taxi derived from OFP offsets
  S.altData.forEach(a => {
    a.currentEta = addMin(baseMin, Math.max(0, a.ofpOffsetMins - taxiMins));
  });
}

_deriveOffZ() — Effective OFF Time

Single source of truth for the effective wheels-off time. Used by renderEnroutePhase(), calcInflightRest(), and renderInflightRestModule().

// Priority chain:
//   1. S.offZ (crew-entered actual OFF)          → source: 'manual'
//   2. S.outZ + S.taxiMins (OFP planned taxi)    → source: 'computed'
//   3. S.outZ alone (no taxi data in OFP)         → source: 'out'
//   4. null — no departure time available

function _deriveOffZ() {
  if (S.offZ && S.offZ.length === 4) return { hhmm: S.offZ, source: 'manual' };
  if (S.outZ && S.outZ.length === 4) {
    if (S.taxiMins) {
      // OUT + taxiMins, with midnight rollover
      const t = (hhmm2min(S.outZ) + S.taxiMins + 14400) % 1440;
      return { hhmm: min2hhmm(t), source: 'computed' };
    }
    return { hhmm: S.outZ, source: 'out' };
  }
  return null;
}
Basis tags: The enroute OFF row and inflight rest module both display a small badge indicating the data source. OUT+Nm = computed (planned), no badge = manual (actual), OUT = OFP has no taxi time. The basis drives the TOC offset arithmetic in calcInflightRest().

WX / NOTAM / Curfew Cascade

When crew edits outZ or offZ, confirmEdit() calls both an ETA recalc function and buildAirportCards(). This re-evaluates weather, NOTAMs, and curfews for all airports against the updated ETAs without requiring the crew to switch tabs.

// confirmEdit() cascade (both branches)
if (_editField === 'outZ' && val.length === 4) {
  recalcAllETAs();
  buildAirportCards();   // ← WX/NOTAM/curfew re-evaluated
}
if (_editField === 'offZ' && val.length === 4) {
  recalcAllETAsFromOff(val);
  buildAirportCards();   // ← WX/NOTAM/curfew re-evaluated
}

GNSS NOTAM Enrichment (evalGNSS)

After evalNotams() classifies GPS/GNSS/RAIM NOTAMs, evalAirport() runs a second pass calling evalGNSS() on each NOTAM. If the GNSS enrichment finds spatial detail (radius, altitude bounds), it appends it inline to the existing GPS issue row. No new rows are created — this avoids duplication with the existing NOTAM classification.

// evalAirport() — GNSS enrichment pass (post evalNotams)
if (typeof evalGNSS === 'function') {
  for (const notam of notamArr) {
    const g = evalGNSS(notam);
    if (!g) continue;
    const parts = [];
    if (g.radius_nm) parts.push('WI ' + g.radius_nm + 'NM');
    if (g.affected_altitudes) parts.push(a.lower + '–' + a.upper);
    if (!parts.length) continue;
    // Append to the first matching GPS/GNSS/RAIM issue row
    const idx = issues.findIndex(i => /GPS|GNSS|RAIM/i.test(i.text));
    if (idx >= 0) issues[idx].text += '<span class="gnss-spatial"> ' + parts.join(' ') + '</span>';
  }
}
// Result: "GPS/GNSS UNRELIABLE — WI 50NM SFC–FL280"

NOTAM Activity Window

Each NOTAM is evaluated against a time window: ORIG uses departure ETA (OUT), DEST/ALT use arrival ETA ± 60 min. The xNW_standalone() parser handles three timestamp formats: 10-digit ICAO (2603052354-2603062354), OFP human (05MAR2354-06MAR2354), and 6-digit (051730-060800). Cross-midnight windows supported.

OFP Inline Schedule Parsing

Atlas/Polar OFPs embed schedules directly below the NOTAM header line. Two formats:

  • Day-range + time window: 08-09 13-17 20-24 1510-2020 — active only on those calendar days, 1510–2020Z
  • Daily time window: 1410-2030 — active every day during that period only

NOTAMs outside their schedule window relative to the flight ETA are suppressed — eliminates false runway closure alarms from overnight maintenance NOTAMs.

Aircraft Applicability Filter

_acftApplies(notamRaw, acftProfile) evaluates wingspan and weight restriction qualifiers. If no qualifier is found, the NOTAM passes (conservative). Wingspan categories: NARROW (<30m), MEDIUM (30–52m), WIDE (>52m, B747/B777).

9. Inflight Rest Module (14 CFR 121.467)

Regulatory basis: 14 CFR 121.467 — Augmented flight crew rest requirements.

calcInflightRest()

Computes the rest window from TOC to pre-landing buffer and divides into blocks.

// Input state:
//   _deriveOffZ()     → effective OFF (manual > computed > out > null)
//   S.tocOffsetMins   → minutes from OFF to TOC (from OFP route table)
//   S.tocZOverride    → crew-entered TOC override (highest priority)
//   S.etaDest         → destination ETA
//   S.crewRestBuffer  → pre-landing buffer in minutes (default 60)
//   S.crewMembers     → crew list; operating count = non-JS, non-DH, active

// TOC arithmetic:
//   source 'manual'  : TOC = offZ + tocOffsetMins
//   source 'computed': TOC = (outZ+taxi) + tocOffsetMins  (taxi already in base)
//   source 'out'     : TOC = outZ + taxiMins + tocOffsetMins  (add taxi explicitly)

// Rest window: TOC → (etaDest − crewRestBuffer)
// totalMins = window duration

// Returns:
{ tocZ, restEndZ, totalMins, operatingCount, tocBasis, blocks3, blocksH, blocksQ }

IndexedDB — PDF Blobs

Imported PDF ArrayBuffers are stored in IndexedDB under 'pb_pdf_store', keyed by flight ID. Accessed by the OFP viewer to render the original PDF canvas. The service worker does not touch IndexedDB.

Service Worker Cache

Two cache namespaces:

  • pb-v1.9.7 — app shell + CDN assets (versioned, pruned on SW activate)
  • pb-share-relay — single-slot relay for Share Target PDF POST (preserved across SW updates)

12. Service Worker & PWA

Cache Strategy

  • Shell pre-cache (atomic): index.html, manifest.json, icon192.png, icon512.png — all must succeed or SW install aborts.
  • CDN pre-cache (best-effort): pdf.js + worker — failure does not abort SW install. Lazy-loader in index.html retries on next online session.
  • Runtime GET strategy: Cache-first; background revalidation for same-origin; 503 text fallback on cache miss + network failure.

Share Target (Web Share Target API)

Declared in manifest.json as POST multipart to ./?share-target. The SW fetch handler intercepts the POST, extracts the ofp_pdf file, writes it to pb-share-relay cache with an X-PB-Filename header, and redirects to /?shared. The _consumeSharedPDF() consumer in index.html reads and deletes the relay entry, then calls processPDF().

SW Update Notification

On activate, the SW posts { type: 'SW_UPDATED', version: CACHE_NAME } to all open window clients. The listener in index.html shows a non-blocking dismissible banner. The crew decides when to reload — the session is never interrupted automatically.

Version bump requirement: Every deployment that changes sw.js must increment CACHE_NAME. Current: pb-v1.9.7. The old cache is pruned by the activate handler; pb-share-relay is explicitly exempted from pruning.

14. ADR-001 Ruleset Architecture

The app follows ADR-001 (Ruleset Modularity). Rulesets are inlined as IIFE-wrapped blocks in index.html:

  • All module-level declarations use var (not const) to avoid block-scoping conflicts
  • 'use strict' stripped before wrapping
  • Browser environment assignments (window.fn = fn) replace module.exports blocks
  • Test harnesses stripped
  • No <script src=""> external tags — single-file constraint
  • console.info not available in Safari/WKWebView — use console.log throughout ruleset code
